00 · Guardrails — read first

Flags logged during design review

1.Emails are split across two <span>s in the mockups (e.g. <span>pat@</span><span>premierpainting.com</span>). This is a mockup-environment workaround only — the preview host rewrites literal email text. In React, render email addresses as plain text. Do not replicate the hack.
2.No notification UI anywhere. No bells, no badges, no unread dots, in any shell, at any breakpoint. The product has no notification feature.
3.Every button maps to a named server action. If a screen needs a new control, it must correspond to an action the system actually performs. The full v1 action set per surface is listed in the keystone notes (§6).
4.v2 wishlist — not in v1, do not build UI for: metric visibility flag ("internal only" metrics — clients see every active metric), view-as-client / "open client portal", client reminder emails, chat, meeting scheduler, global search.
5.Amber contrast rule (locked): white text only on amber-700+. Amber primary buttons = white on #B45309, hover #92400E. Amber-600 #D97706 is reserved for fills, progress, ticks, active states and focus rings — never under white text. Amber text on light backgrounds uses 700/800.
6.Competitor data is schema-only: name/handle, niche, priority, follower count (nullable), avg views (nullable), top format, hook style, what's-working, the-gap, our-play. Nullable fields hide. No citations counts, ratings, map-pack data, deltas, or "You" comparison rows.

06 · Per-keystone engineering notes

Behaviors that aren't visible in a screenshot

K1 · Login — auth shell
  • Dark split card on --dark-bg with --dark-glow; form panel 480px, brand panel fills. Mobile: brand panel collapses to a compact glowing header above the form.
  • Auth error: generic message ("That email and password don't match. Try again, or reset your password."), both fields flag red — the system can't say which credential failed. Typing clears the error.
  • Links on dark use amber-400 #FBBF24; CTA is white on amber-700.
  • Forgot-password, accept-invite (with password-strength hint), and the confirm/expired-link interstitials reuse this shell as a single centered card.
  • Support contact: team@fourpielabs.com (plain text in production — see guardrail 1).
K2 · Client Dashboard — client shell, spacious
  • Shell: top nav (logo · 7 items · avatar). Active item = --surface-2 pill. Mobile: bottom tab bar — 5 slots (Home, Program, Content, Numbers, More-sheet), active = 48px charcoal circle, white pill bar floats 14px from edges.
  • KPI card: overline label + month · Bricolage numeral · delta pill (▲▼ + value) · "vs N last month". Honest down-deltas stay red — never hide them.
  • Checklist: client rows tick with the 300ms spring pop (amber-600 fill); team rows show "We'll take care of this." and count as done; link rows get a charcoal pill button. When all complete → collapses to the single-line green "Onboarding complete — you're all set." with a Review affordance to reopen.
  • Roadmap: 4 milestone columns, top border = status color (Done amber-700 / current amber-400 / upcoming border), "Day N of 90" chip, 6px progress bar.
  • Report card uses the dark surface + glow treatment — the one dark card on the page.
K3 · Client Performance
  • Chart card: metric selector (select on desktop, pill segmented control on mobile) swaps the series; axis re-scales; window = months with data only.
  • Monthly table: newest first, current month row tinted #FFFBEB, per-cell ▲▼ vs prior, first month gets a "first full month" note and no deltas. Mobile: collapses to per-month cards (3 key metrics + load more).
  • Competitor cards: schema fields only (guardrail 6); qualitative intel leads; "Our play" gets the amber-50 block. Nullable follower/views hide the stat row entirely.
  • Reports: list (selected row = amber-50 tint + 3px amber-600 left border) + viewer (markdown prose style, Download PDF). Clients never see drafts — staff-only state.
K4 · Staff Client Overview — staff shell, compact
  • Shell: 264px sidebar (logo, client switcher, nav w/ charcoal active pill, user row). Tablet 768–1024: icon rail. Mobile: drawer + client-switcher chip in the top bar; tab strip scrolls horizontally; no bottom tab bar on staff.
  • Client header: initials tile (amber-100/amber-800, real logo when provided), name + program chip + status chip, partner/day meta, ⋯ actions.
  • Tab bar: underline style; beyond ~8 tabs overflow into "More" (holds Notes · Reports · Updates · Files · Settings).
  • "Waiting on client" banner derives from deliverables in needs_review; its action is Open deliverable (navigation, not email).
  • Overview cards: checklist ring (79%) + per-phase counts; latest-metrics 2×2 with "Entered [date] by [user]" + Enter-month CTA; activity feed = read view of the audit trail.
K5 · Staff Metrics
  • Entry grid: native inputs, DOM order = visual column order so Tab flows straight down; right-aligned tabular numerals; dirty rows tint --unsaved-bg + amber-400 input border; validation is per-field and inline (e.g. "Must be between 0 and 5") and blocks Save for that field only.
  • Sticky save bar lives inside the scroll container: "[n] unsaved · [n] error" + Discard + Save [Month]. Save = bulk upsert for the selected month.
  • Definitions: name + unit only (no visibility flag — guardrail 4); drag to reorder; Add/Edit are the only mutations.
  • CSV import: dropzone → preview with per-row OK/error and specific messages → "Import N rows" commits clean rows only. Framed as pre-portal history backfill.
  • Client preview chart = the same Recharts component the client sees; entry-status card lists recent months In progress/Complete.
K6 · Admin Users
  • Action set (1:1 with server actions): invite · resend · revoke · deactivate · reactivate · assign client. Nothing else.
  • Invite form: email + role select; client select appears and is required only when role = Client.
  • Pending rows: warm tint + dashed avatar + dashed chip, Resend + Revoke inline. Revoke and Deactivate both get confirm dialogs ("They'll be signed out and blocked at next request. You can reactivate anytime.").
  • Self-lockout protection: the signed-in admin's row shows a "You" marker and no deactivate control.
  • Rate-limit toast copy is config-agnostic: "Rate limit reached — try again in a few minutes." Error toasts name the cause (e.g. "SMTP authentication failed") and offer Retry only when retrying can work.
  • Inactive = strikethrough-feel + 0.6 opacity, never red; Reactivate is the row action.
