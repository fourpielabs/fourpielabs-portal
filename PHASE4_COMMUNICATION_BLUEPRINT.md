# Phase 4 — Communication + Notifications (locked blueprint)

> Built AFTER onboarding/projects. Shares infrastructure with notifications,
> so the two are built together. Three product decisions are CONFIRMED below.
> Same discipline as the auth phase: plan mode, migration hard-stop before any
> db push, test:rls re-run to a new documented baseline, new RLS tests for
> every new access path.

## Confirmed product decisions
1. TWO thread types per client:
   - **Client↔agency shared thread** — one per client; the client, their
     assigned team members, and admins all participate.
   - **Internal staff-only thread** — one per client; team (assigned) + admin
     only; the client can NEVER see it (enforced at RLS, not just UI).
2. **Client-initiated allowed** — a client can start/post in their shared
   thread anytime (not reply-only). This is a new client-write path.
3. **Email notifications: YES** — new-message and key-event emails via the
   existing Resend SMTP, so messaging actually gets used.

## Core model — messages attached to work, not free chat
Conversations live as threads scoped to a client (the Asana pattern: context
never lost), NOT a Slack-style global chat. Deliberately NOT building in v1:
no global chat, no individual DMs, no voice/video, no presence/typing.

## Data model (new tables, RLS on each)
- `threads`: id, client_id, type (`client_shared` | `internal`), created_at.
  (Seed one of each per client on client creation, like the checklist/roadmap
  seeding precedent.)
- `messages`: id, thread_id, client_id (denormalized for RLS), author_id,
  body (markdown), created_at, edited_at. Optional: attachment_path (reuse the
  existing files storage + signed-URL pattern), context_type/context_id
  (link a message to a deliverable or report — the "attached to work" value).
- `message_reads` (or last_read_at per user per thread): drives unread counts.
- `notifications`: id, user_id, type, title, body, link, read_at, created_at.
  Generated by real events: new message, deliverable delivered, report
  published, deliverable approved (ties into the client-write path just built),
  project status changed.

## Access control (the security shape — mirrors existing patterns)
- **threads/messages SELECT**:
  - admin: all.
  - team: where `is_assigned(client_id)` (both thread types).
  - client: where `client_id = my_client_id() AND thread.type = 'client_shared'`
    — the internal thread is invisible at the DB layer.
- **message INSERT (the new client-write path)**: via a SECURITY DEFINER RPC
  (the `toggle_checklist_item` / `set_deliverable_approval` precedent), NOT a
  direct client UPDATE/INSERT policy — preserves the invariant. The RPC
  validates: authenticated, the thread belongs to the caller's allowed scope
  (client → only their own `client_shared` thread; team → only assigned
  clients' threads; admin → any), author_id = caller, body non-empty. Clients
  physically cannot post to the internal thread or another client's thread.
- **notifications**: user reads/updates only their own rows; inserts via
  service-role server actions on events (the audit_log precedent).

## Email notifications (Resend — already configured)
- On new message in the client_shared thread → email the OTHER side
  (client posts → notify assigned team/admin; team posts → notify the client).
- On internal-thread message → notify only the other staff participants.
- Branded, plain-text-safe, links to the thread. Throttle/batch so a burst of
  messages doesn't spam (e.g. one digest per short window). No message BODY in
  the email if it risks leaking sensitive content — link to the portal.

## UI
- A messaging surface per client: client side = their shared thread (a clean
  conversation view + composer); staff side = both threads (tabbed: "Client"
  and "Internal") within the per-client workspace.
- The notification bell — the ONE place a bell is legitimate now, because a
  real feature backs it: unread count, dropdown list, mark-read, deep-link.
  (Until this phase, "no notification UI" was the rule precisely because there
  was no feature behind it.)
- Realtime via Supabase Realtime (built-in — no separate websocket server) so
  new messages appear live.

## New RLS tests (added to the suite — new baseline)
- client CAN post to own shared thread (via RPC); CANNOT post to internal
  thread; CANNOT post to another client's thread; CANNOT direct-INSERT
  messages (no policy — denied).
- client CANNOT read internal thread (0 rows) or another client's threads.
- team CAN read/post both thread types for ASSIGNED clients only; denied for
  unassigned.
- notifications: user reads only own; cannot read another user's.
- anon: everything denied.

## Build order (each: plan → migration hard-stop → push → test:rls → verify)
4a. Schema + RLS + seeding + the message-post RPC + new RLS tests (hard stop
    for migration review, like the auth phase).
4b. Notifications table + the bell UI + event triggers (deliverable delivered/
    approved, report published).
4c. Messaging UI (client shared thread; staff dual-thread) + Realtime.
4d. Email notifications via Resend (with throttle/batch).
